## 1. 核心理念

### 1.1 基本原则

| 原则 | 说明 |
|------|------|
| **KISS** | Keep It Simple, Stupid — 简洁至上，避免过度工程化 |
| **第一性原理** | 立足根本剖析问题，不盲从惯例 |
| **事实为本** | 以事实为最高准则，有误必纠 |
| **渐进式开发** | 多轮迭代，先调研再设计再编码 |

### 1.2 开发流程

```
需求分析 → 方案设计 → 评审确认 → 任务分解 → 编码实现 → 测试验证
```

## 2. 架构分层

### 2.1 三层架构

```
┌─────────────────────────────────────────────────┐
│                   Board 层                       │
│         (板级服务：bsp_lcd, bsp_key...)          │
├─────────────────────────────────────────────────┤
│                   Driver 层                      │
│         (设备驱动：dev_st7789, dev_w25q...)      │
├─────────────────────────────────────────────────┤
│                  Interface 层                    │
│         (硬件抽象：port_spi, port_i2c...)        │
└─────────────────────────────────────────────────┘
```

### 2.2 层级职责

| 层级 | 命名前缀 | 职责 | 依赖方向 |
|------|----------|------|----------|
| **Board** | `bsp_` | 板级业务封装，面向应用 | 依赖 Driver |
| **Driver** | `dev_` | 设备驱动实现，芯片级封装 | 依赖 Interface |
| **Interface** | `port_` | 硬件抽象层，屏蔽 MCU 差异 | 依赖 HAL/LL |

### 2.3 依赖规则

- **上层可依赖下层**，下层禁止依赖上层
- **同层模块禁止循环依赖**
- 跨层调用只能通过公开接口

## 3. 命名规范

### 3.1 文件命名

| 类型 | 格式 | 示例 |
|------|------|------|
| Board 层 | `bsp_<功能>.c/h` | `bsp_lcd.c`, `bsp_key.c` |
| Driver 层 | `dev_<芯片>.c/h` | `dev_st7789.c`, `dev_w25q.c` |
| Interface 层 | `port_<外设>.c/h` | `port_spi.c`, `port_i2c.c` |

### 3.2 函数命名

| 层级 | 格式 | 示例 |
|------|------|------|
| Board 层 | `bsp_<模块>_<动作>` | `bsp_lcd_init()`, `bsp_key_scan()` |
| Driver 层 | `<芯片>_<动作>` 或统一前缀 | `lcd_init()`, `w25q_read()` |
| Interface 层 | `port_<外设>_<动作>` | `port_spi_write()`, `port_i2c_read()` |
| 私有函数 | `static` + `s_` 前缀或模块前缀 | `static void st7789_write_cmd()` |

### 3.3 变量命名

| 类型 | 格式 | 示例 |
|------|------|------|
| 局部变量 | `snake_case` | `uint16_t pixel_count;` |
| 全局变量 | `g_` 前缀 | `uint8_t g_system_state;` |
| 静态变量 | `s_` 前缀 | `static uint16_t s_lcd_width;` |
| 结构体成员 | `snake_case` | `.pin_id`, `.buffer_size` |
| 指针变量 | `p_` 前缀或 `*` 靠变量 | `uint8_t *p_data;` 或 `uint8_t *data;` |

### 3.4 宏与枚举命名

| 类型 | 格式 | 示例 |
|------|------|------|
| 宏定义 | 全大写 + 下划线 | `#define LCD_WIDTH 240` |
| 带参数宏 | 全大写 + 括号包裹 | `#define MAX(a, b) ((a) > (b) ? (a) : (b))` |
| 枚举值 | 大写 + 模块前缀 | `PORT_SPI_1`, `BSP_OK`, `LCD_COLOR_RED` |
| 枚举类型 | `xxx_t` 后缀 | `typedef enum { ... } port_spi_id_t;` |

### 3.5 宏定义格式规范

#### 3.5.1 括号使用规则

| 场景 | 括号要求 | 原因 |
|------|----------|------|
| 函数式宏参数 | **必须** | 避免运算优先级问题 |
| 函数式宏整体 | **必须** | 避免上下文运算优先级问题 |
| 表达式宏 | **必须** | 同上 |
| 简单单值宏 | 推荐 | 统一风格，避免遗漏 |

```c
/* 正确：函数式宏参数和整体都加括号 */
#define MAX(a, b)         ((a) > (b) ? (a) : (b))
#define MIN(a, b)         ((a) < (b) ? (a) : (b))
#define ALIGN_UP(x, n)    (((x) + (n) - 1) & ~((n) - 1))
#define BIT_SET(reg, bit) ((reg) |= (1U << (bit)))
#define BIT_CLR(reg, bit) ((reg) &= ~(1U << (bit)))

/* 错误：缺少括号导致潜在问题 */
#define SQUARE(x)         (x * x)           // SQUARE(1+2) = 1+2*1+2 = 5，预期 9
#define DOUBLE(x)         x + x             // DOUBLE(1)*2 = 1+1*2 = 3，预期 4

/* 正确：表达式宏加括号 */
#define BUFFER_SIZE       (4096)
#define TIMEOUT_MS        (1000U)

/* 可接受：简单单值宏（但推荐统一加括号） */
#define CMD_NOP           0x00
#define CMD_RESET         0x01

/* 推荐：统一加括号 */
#define CMD_NOP           (0x00U)
#define CMD_RESET         (0x01U)
```

#### 3.5.2 U 后缀使用规则

**MISRA C:2012 Rule 7.2**：无符号整数常量应加 `U` 后缀。

| 场景 | U 后缀 | 原因 |
|------|--------|------|
| 位掩码 | **必须** | 避免符号扩展陷阱 |
| 寄存器值/命令 | **推荐** | 嵌入式常见陷阱 |
| 移位操作数 | **必须** | 避免移位溢出 |
| 普通计数/尺寸 | 可选 | 风险较低 |
| 枚举值 | 可选 | 类型已明确 |

```c
/* 必须加 U：位掩码 */
#define ST7789_MADCTL_MY       (0x80U)
#define ST7789_MADCTL_MX       (0x40U)
#define ST7789_MADCTL_BGR      (0x08U)
#define ENABLE_MASK            (0x01U)

/* 推荐加 U：寄存器命令 */
#define ST7789_CMD_NOP         (0x00U)
#define ST7789_CMD_SWRESET     (0x01U)
#define ST7789_CMD_SLPIN       (0x10U)

/* 必须加 U：移位操作 */
#define BIT(n)                 (1U << (n))
#define FLAG_ENABLE            (1U << 0)
#define FLAG_BUSY              (1U << 1)

/* 可选：普通常量 */
#define BUFFER_SIZE            (4096)
#define MAX_RETRY              (3)
#define LCD_WIDTH              240
#define LCD_HEIGHT             320

/* 危险示例：不加 U 导致的问题 */
int result = 0x80 << 24;      // 危险！0x80 是 int，左移可能溢出
int result = 0x80U << 24;     // 安全：0x80U 是 unsigned，无符号扩展

/* 枚举值：类型已明确，U 后缀可选 */
typedef enum
{
    STATE_IDLE = 0,
    STATE_RUNNING = 1,
    STATE_ERROR = 2
} state_t;
```

#### 3.5.3 推荐的统一风格

```c
/* 寄存器命令/位掩码：括号 + U 后缀 */
#define ST7789_CMD_NOP         (0x00U)
#define ST7789_CMD_SWRESET     (0x01U)
#define ST7789_MADCTL_MY       (0x80U)
#define ST7789_MADCTL_BGR      (0x08U)

/* 普通常量：括号可选，U 后缀可选 */
#define ST7789_BUFFER_SIZE     (4096)
#define ST7789_WIDTH           240
#define ST7789_HEIGHT          320
#define TIMEOUT_MS             (1000U)

/* 函数式宏：参数括号 + 整体括号 + U 后缀（如涉及位操作） */
#define MAX(a, b)              ((a) > (b) ? (a) : (b))
#define MIN(a, b)              ((a) < (b) ? (a) : (b))
#define BIT(n)                 (1U << (n))
#define ARRAY_SIZE(arr)        (sizeof(arr) / sizeof((arr)[0]))
```

#### 3.5.4 多语句宏

使用 `do { ... } while(0)` 包裹，确保宏可以像函数一样使用：

```c
/* 正确：多语句宏使用 do-while(0) */
#define SAFE_FREE(ptr) \
    do { \
        if ((ptr) != NULL) { \
            free(ptr); \
            (ptr) = NULL; \
        } \
    } while(0)

/* 使用时可以正常加分号 */
SAFE_FREE(buffer);  // 展开后语法正确

/* 错误：使用逗号或分号会导致问题 */
#define SAFE_FREE(ptr) free(ptr), ptr = NULL  // if 条件下行为异常
```

## 4. 代码格式

### 4.1 缩进与空格

| 项目 | 规则 |
|------|------|
| 缩进 | 4 空格，禁止 Tab |
| 大括号 | Allman 风格（独占一行） |
| 运算符两侧 | 加空格：`a = b + c` |
| 逗号后 | 加空格：`func(a, b, c)` |
| 分号后 | 加空格：`for (i = 0; i < 10; i++)` |

### 4.2 函数定义

```c
/* 正确：紧凑单行 */
bsp_status_t port_spi_write(port_spi_id_t bus, uint8_t *data, uint16_t len)
{
    /* 函数体 */
}

/* 错误：禁止人为换行对齐 */
bsp_status_t port_spi_write(port_spi_id_t bus,
                             uint8_t      *data,
                             uint16_t      len)
{
    /* 函数体 */
}
```

### 4.3 控制语句

```c
/* if 语句 */
if (condition)
{
    /* 分支 */
}
else
{
    /* 分支 */
}

/* for 循环 */
for (uint32_t i = 0; i < count; i++)
{
    /* 循环体 */
}

/* while 循环 */
while (is_busy)
{
    /* 循环体 */
}

/* switch 语句 */
switch (state)
{
    case STATE_IDLE:
        handle_idle();
        break;

    case STATE_RUNNING:
        handle_running();
        break;

    default:
        handle_error();
        break;
}
```

### 4.4 单行长度

- **最大 120 字符**
- 超长行在低优先级运算符处换行，运算符置于新行行首

```c
bsp_status_t ret = port_spi_tx_dma(bus,
                                   buffer,
                                   length,
                                   callback,
                                   user_data);
```

### 4.5 空行规则

| 位置 | 规则 |
|------|------|
| 函数之间 | 1 个空行 |
| 逻辑块之间 | 1 个空行 |
| 头文件包含块后 | 1 个空行 |
| 宏定义块后 | 1 个空行 |
| 结构体/枚举定义后 | 1 个空行 |

## 5. 注释规范

### 5.1 文件头注释

```c
/**
 * @file port_spi.c
 * @brief SPI 接口层实现
 * @note 封装 HAL SPI 接口，提供统一抽象
 */
```

### 5.2 函数注释（Doxygen 格式）

```c
/**
 * @brief 初始化 SPI 总线
 * @param bus SPI 总线 ID
 * @return bsp_status_t 执行结果
 *         - BSP_OK 成功
 *         - BSP_EINVAL 参数无效
 *
 * @note 调用前需确保 GPIO 已初始化
 *
 * @example
 *   bsp_status_t ret = port_spi_init(PORT_SPI_1);
 *   if (ret != BSP_OK) {
 *       // 错误处理
 *   }
 */
bsp_status_t port_spi_init(port_spi_id_t bus);
```

### 5.3 注释风格选择

| 场景 | 风格 | 示例 |
|------|------|------|
| 文件头、函数接口 | Doxygen `/** */` | 见上方示例 |
| 代码块说明 | 块注释 `/* */` | `/* 1. 初始化硬件 */` |
| 单行说明 | 行注释 `//` | `// 等待 DMA 完成` |
| 枚举值说明 | 行尾注释 `//` | `PORT_SPI_1,  // SPI1` |

### 5.4 注释语言

- **中文为主**：便于理解，言简意赅
- **英文为辅**：通用术语、寄存器名称

### 5.5 注释禁忌

- **禁止注释掉的代码**：删除而非注释
- **禁止冗余注释**：代码本身应清晰
- **禁止误导性注释**：注释需与代码同步更新

## 6. 头文件规范

### 6.1 头文件保护

```c
#ifndef __PORT_SPI_H
#define __PORT_SPI_H

#ifdef __cplusplus
extern "C" {
#endif

/* 内容 */

#ifdef __cplusplus
}
#endif

#endif /* __PORT_SPI_H */
```

### 6.2 包含顺序

```c
/* 1. 系统头文件 */
#include <stdint.h>
#include <stdbool.h>
#include <string.h>

/* 2. 项目公共头文件 */
#include "bsp_board.h"
#include "bsp_status.h"

/* 3. 模块内部头文件 */
#include "port_gpio.h"
```

### 6.3 头文件原则

| 原则 | 说明 |
|------|------|
| 自包含 | 头文件能独立编译 |
| 最小暴露 | 只暴露公开接口，隐藏实现细节 |
| 禁止定义变量 | 头文件只声明，不定义 |
| 禁止循环依赖 | A 依赖 B，则 B 不能依赖 A |

### 6.4 外部依赖包含规范

**原则**：每个源文件自行包含所需的外部库头文件，禁止集中包含在公共头文件中。

**外部库定义**：
- HAL 库（`stm32f4xx_hal.h`）
- CMSIS 库
- 第三方中间件（FreeRTOS、FatFS 等）

**规范说明**：

| 做法 | 是否允许 | 原因 |
|------|----------|------|
| 在 .c 文件中直接包含 HAL | ✅ 推荐 | 依赖清晰，编译隔离 |
| 在 bsp_board.h 中包含 HAL | ❌ 禁止 | 编译依赖膨胀，职责模糊 |
| 创建 bsp_hal_conf.h 集中管理 | ⚠️ 可选 | 适用于大型项目 |

**正确示例**：

```c
/* port_tick.c */
#include "port_tick.h"
#include "stm32f4xx_hal.h"  // 自己负责外部依赖

void port_tick_init(void)
{
    /* HAL 库已初始化 SysTick */
}

uint32_t port_tick_get_ms(void)
{
    return HAL_GetTick();
}
```

**错误示例**：

```c
/* bsp_board.h */
#include "stm32f4xx_hal.h"  // ❌ 禁止！所有包含 bsp_board.h 的文件都被迫依赖 HAL

/* port_tick.c */
#include "bsp_board.h"      // 间接获得 HAL，依赖不透明
```

**原因分析**：

1. **编译效率**：HAL 头文件修改时，只重编译直接依赖的文件
2. **职责清晰**：bsp_board.h 是 BSP 内部核心，不应绑定特定 HAL 库
3. **最小依赖**：每个文件只包含实际需要的头文件

## 7. 错误处理

### 7.1 状态码定义

```c
typedef enum
{
    BSP_OK       =  0,  // 成功
    BSP_ERROR    = -1,  // 通用错误
    BSP_EINVAL   = -2,  // 参数无效
    BSP_ENOMEM   = -3,  // 内存不足
    BSP_ETIMEOUT = -4,  // 超时
    BSP_BUSY     = -5,  // 资源忙
} bsp_status_t;
```

### 7.2 错误处理模式

```c
/* 模式 1：提前返回 */
bsp_status_t func(uint8_t *data, uint16_t len)
{
    if (data == NULL) return BSP_EINVAL;
    if (len == 0) return BSP_EINVAL;

    /* 正常逻辑 */
    return BSP_OK;
}

/* 模式 2：错误传递 */
bsp_status_t outer_func(void)
{
    bsp_status_t ret = inner_func();
    if (ret != BSP_OK)
    {
        return ret;  // 向上传递错误
    }

    return BSP_OK;
}
```

### 7.3 参数检查

```c
bsp_status_t port_i2c_read(port_i2c_id_t bus, uint8_t addr, uint8_t *data, uint16_t len)
{
    /* 1. 边界检查 */
    if (bus >= PORT_I2C_MAX) return BSP_EINVAL;

    /* 2. 空指针检查 */
    if (data == NULL) return BSP_EINVAL;

    /* 3. 逻辑检查 */
    if (len == 0) return BSP_EINVAL;

    /* 正常逻辑 */
    return BSP_OK;
}
```

## 8. 安全规范

### 8.1 内存安全

| 规则 | 说明 |
|------|------|
| 禁止数组越界 | 访问前检查索引范围 |
| 禁止内存泄漏 | 动态分配需对应释放 |
| 禁止野指针 | 释放后置 NULL |
| 禁止未初始化使用 | 变量声明时初始化 |

### 8.2 并发安全

| 规则 | 说明 |
|------|------|
| 全局变量保护 | 中断/多线程访问需加锁或 volatile |
| 避免优先级反转 | 合理设置中断优先级 |
| 避免死锁 | 按固定顺序获取锁 |

### 8.3 硬件安全

| 规则 | 说明 |
|------|------|
| 超时保护 | 所有阻塞操作需有超时机制 |
| 错误恢复 | 外设错误需有恢复策略 |
| 硬件验证 | 调试时优先排查硬件问题 |

## 10. 代码示例

### 10.1 Interface 层示例

```c
/**
 * @file port_spi.h
 * @brief SPI 接口层头文件
 * @note 提供统一的 SPI 操作接口
 */

#ifndef __PORT_SPI_H
#define __PORT_SPI_H

#include <stdint.h>
#include "bsp_board.h"

/* SPI 总线 ID 枚举 */
typedef enum
{
    PORT_SPI_1 = 0,  // SPI1
    PORT_SPI_2,      // SPI2
    PORT_SPI_MAX     // 计数哨兵
} port_spi_id_t;

/* DMA 回调函数类型 */
typedef void (*port_spi_dma_cb_t)(bool success, void *user_data);

/* 公开 API */
bsp_status_t port_spi_init(port_spi_id_t bus);
bsp_status_t port_spi_write(port_spi_id_t bus, uint8_t *data, uint16_t len);
```

```c
/**
 * @file port_spi.c
 * @brief SPI 接口层实现
 */

#include "port_spi.h"
#include "spi.h"

/* 硬件句柄映射 */
static SPI_HandleTypeDef *s_spi_map[] = {
    [PORT_SPI_1] = &hspi1,
    [PORT_SPI_2] = &hspi2,
};

/**
 * @brief 初始化 SPI 总线
 */
bsp_status_t port_spi_init(port_spi_id_t bus)
{
    if (bus >= PORT_SPI_MAX) return BSP_EINVAL;

    SPI_HandleTypeDef *hspi = s_spi_map[bus];
    if (hspi == NULL) return BSP_ERROR;

    /* HAL 初始化由 CubeMX 完成，此处仅验证状态 */
    if (hspi->State == HAL_SPI_STATE_RESET) return BSP_ERROR;

    return BSP_OK;
}

/**
 * @brief 阻塞写入数据
 */
bsp_status_t port_spi_write(port_spi_id_t bus, uint8_t *data, uint16_t len)
{
    if (bus >= PORT_SPI_MAX) return BSP_EINVAL;
    if (data == NULL || len == 0) return BSP_EINVAL;

    SPI_HandleTypeDef *hspi = s_spi_map[bus];
    if (hspi == NULL) return BSP_ERROR;

    HAL_StatusTypeDef ret = HAL_SPI_Transmit(hspi, data, len, 1000);

    return (ret == HAL_OK) ? BSP_OK : BSP_ERROR;
}
```

## 11. Git 提交消息规范

### 11.1 消息结构

```
<type>(<scope>): <subject>

[可选 body]

[可选 footer]
```

### 11.2 标题行（必填）

**格式**：`<type>(<scope>): <中文简述>`

**type 类型**：

| 类型 | 说明 | 示例 |
|------|------|------|
| `feat` | 新功能 | 新增驱动、接口、Demo |
| `fix` | Bug 修复 | 修复 DMA 竞态、参数校验遗漏 |
| `refactor` | 重构（不改变功能） | 代码结构优化、命名统一 |
| `docs` | 文档变更 | 更新 README、代码规范 |
| `test` | 测试相关 | 新增测试用例、测试框架 |
| `chore` | 构建/工具变更 | 更新 .clang-format、Makefile |
| `style` | 代码格式（不影响逻辑） | 格式化、缩进修正 |
| `perf` | 性能优化 | 算法优化、内存占用降低 |

**scope 范围**：

| 范围 | 说明 |
|------|------|
| `arch` | 架构、目录结构、规范 |
| `interface` | Interface 层（port_spi, port_i2c...） |
| `driver` | Driver 层（dev_st7789, dev_w25q...） |
| `board` | Board 层（bsp_lcd, bsp_key...） |
| `middleware` | 中间件（Letter Shell, EasyLogger...） |
| `demo` | Demo 示例 |
| `app` | 应用层任务 |

**subject 要求**：
- 使用中文简述
- 不超过 50 字符
- 不以句号结尾
- 使用祈使语气（"新增"而非"新增了"）

### 11.3 正文（可选）

**格式要求**：
- 与标题空一行
- 每行不超过 72 字符
- 说明 **做了什么**、**为什么做**、**影响范围**

**内容模板**：

```markdown
<type>(<scope>): <subject>

变更内容：
- 新增 xxx 功能
- 修正 xxx 问题
- 重构 xxx 模块

影响范围：
- 影响模块 A、模块 B
- 需要同步更新 xxx 配置

技术细节：
- 使用 xxx 方案解决 xxx 问题
- 参考 [链接/文档]
```

### 11.4 Footer（可选）

**用途**：
- **Breaking Changes**：不兼容变更说明
- **Issue 引用**：关联 Issue 编号
- **验收命令**：Shell 验收命令

**格式**：

```markdown
BREAKING CHANGE: <不兼容变更说明>

Closes #123
Refs #456

验收: shell> version
```

### 11.5 完整示例

**示例 1：新功能（简单）**

```
feat(interface): port_gpio - GPIO 逻辑映射层
```

**示例 2：新功能（带正文）**

```
feat(driver): dev_ws2812 - PWM+DMA RGB LED 驱动

变更内容：
- 新增 WS2812B RGB LED 驱动
- 支持 RGB565 颜色格式
- 实现彩虹渐变、呼吸灯效果

技术细节：
- 使用 TIM PWM+DMA 方案，CPU 占用率 < 1%
- 修正 V1 颜色宏复合字面量兼容问题

验收: shell> rgb_rainbow
```

**示例 3：Bug 修复**

```
fix(driver): dev_ws2812 - DMA 传输完成标志竞态

问题：
DMA 传输完成回调中直接修改状态标志，与主循环检测存在竞态，
导致偶发性颜色错乱。

修复：
- 使用 volatile 修饰状态标志
- 回调中通过队列通知主循环
- 增加传输超时保护

验收: shell> rgb_test 100  // 连续运行 100 次无异常
```

**示例 4：不兼容变更**

```
refactor(interface): port_spi - 统一返回值类型

变更内容：
- 所有 API 返回值从 void 改为 bsp_status_t
- 新增参数校验

BREAKING CHANGE: 
port_spi_write() 等函数现在返回 bsp_status_t，
调用方需处理返回值。

Closes #42
```

**示例 5：重构**

```
refactor(driver): dev_st7789 - 代码规范化

变更内容：
- 统一命名风格为 snake_case
- 宏定义添加括号和 U 后缀
- 补充 Doxygen 注释

影响范围：
- 对外 API 不变，仅内部重构
- bsp_lcd.c 需同步更新头文件包含
```

### 11.6 提交粒度建议

| 粒度 | 说明 | 示例 |
|------|------|------|
| **单功能单提交** | 一个提交只做一件事 | 新增驱动、修复 Bug 分开提交 |
| **接口→驱动→Demo 分离** | 同一总线分三个提交 | port_spi / dev_st7789 / demo_lcd |
| **可编译可验收** | 每个提交都能编译通过 | 禁止中间态提交 |

### 11.7 禁止事项

| 禁止 | 原因 |
|------|------|
| `update`、`fix bug` 等模糊描述 | 无法追溯变更内容 |
| 一个提交包含多个不相关功能 | 难以回滚和审查 |
| 提交无法编译的代码 | 破坏主分支可用性 |
| 提交敏感信息 | 安全风险 |
